# Environment Variable Validation Guide

## Overview

The Livin backend now includes a comprehensive environment variable validation system that runs at startup. This ensures that all required configuration is present before attempting to start the Express server or connect to the database.

## Key Features

✓ **Early validation** - Checks all required variables before the app starts  
✓ **Clear error messages** - Explains what's missing and how to fix it  
✓ **Production-aware** - Different error handling for development vs production  
✓ **Masked secrets** - Logs show only first/last 4 characters of sensitive values  
✓ **Helpful hints** - Provides inline commands to generate secrets when needed

## Required Variables

| Variable       | Purpose                      | Min Length | Example                               |
| -------------- | ---------------------------- | ---------- | ------------------------------------- |
| `DATABASE_URL` | PostgreSQL connection string | -          | `postgresql://user:pass@host:5432/db` |
| `JWT_SECRET`   | Auth token signing secret    | 32 chars   | Generated by crypto                   |

## Optional Variables

| Variable                 | Default                                       | Purpose                           |
| ------------------------ | --------------------------------------------- | --------------------------------- |
| `PORT`                   | `5000`                                        | HTTP port (Railway sets this)     |
| `NODE_ENV`               | `development`                                 | Set to `production` in production |
| `CORS_ORIGIN`            | `http://localhost:5173,http://localhost:3000` | Allowed frontend origins          |
| `ADMIN_INITIAL_PASSWORD` | `admin123`                                    | Initial admin password            |
| `BACKEND_URL`            | (unset)                                       | Absolute backend URL for images   |

## Startup Validation Output

When the server starts, you'll see:

```
[STARTUP] Environment configuration validated:
  NODE_ENV: production
  PORT: 5000
  DATABASE: postgresql://host:5432/railway
  JWT_SECRET: 42d5...d26

  BACKEND_URL: https://backend.up.railway.app

[OK] Server running on port 5000
[OK] Database: Prisma client initialized
[OK] Admin user: Ready
```

## Error Handling

If required variables are missing, the server immediately exits with clear instructions:

```
[FATAL] Missing or invalid required environment variables in Railway:

  - DATABASE_URL: Missing (required in production)
     PostgreSQL connection string. Example: postgresql://user:password@host:5432/dbname
  - JWT_SECRET: Missing (required in production)
     256-bit hex string used to sign auth tokens.

To fix this:
  1. Go to your Railway project dashboard
  2. Navigate to Variables
  3. Add each missing variable from the list above
  4. Redeploy the service
```

## Railway Deployment Checklist

1. **Set required variables in Railway Dashboard:**
   - Go to your service → Variables
   - Add `DATABASE_URL` from PostgreSQL plugin
   - Add `JWT_SECRET` (generate if needed)

2. **Set production variables:**
   - `NODE_ENV=production`
   - `ADMIN_INITIAL_PASSWORD=<your-strong-password>`
   - `CORS_ORIGIN=<your-frontend-domain>`
   - `BACKEND_URL=<your-railway-backend-url>`

3. **Deploy and check logs:**
   - Watch the build/deployment logs
   - You should see `[STARTUP] Environment configuration validated`
   - Verify it shows the correct values (with masked secrets)

4. **Health check:**
   - Visit `https://your-backend.up.railway.app/health`
   - Should return: `{"status":"ok","environment":"production",...}`

## Generating JWT_SECRET

If you need to generate a new JWT_SECRET:

```bash
node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"
```

Or in Railway's build logs (if you need to do it there):

```bash
node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"
```

## Local Development Setup

1. **Copy env example:**

   ```bash
   cd server
   cp .env.example .env
   ```

2. **Edit .env with your values:**

   ```bash
   DATABASE_URL="postgresql://localhost:5432/livin"
   JWT_SECRET="your-generated-secret-here"
   ```

3. **Start the app:**
   ```bash
   npm run dev
   ```

You should see the validation output immediately.

## Troubleshooting

### `DATABASE_URL is missing or invalid`

**Solution:**

- In Railway: Add the `DATABASE_URL` from your PostgreSQL plugin
- Locally: Ensure PostgreSQL is running and the connection string is correct

### `JWT_SECRET: Too short`

**Solution:**

- Your secret is less than 32 characters
- Generate a new one with the command above
- Must be at least 32 characters for security

### `CORS_ORIGIN: Origin not allowed`

Browser shows: `Access to XMLHttpRequest has been blocked by CORS policy`

**Solution:**

- Check that your frontend domain is in `CORS_ORIGIN`
- Multiple domains should be comma-separated: `https://app.com,https://admin.com`
- No trailing spaces: `https://app.com, https://admin.com` is invalid

## Code Integration

The validation system is implemented in `src/lib/env-validation.ts` and is called at:

1. **Module load time** in `src/index.ts` - validates before starting Express
2. **Module load time** in `src/lib/prisma.ts` - validates before creating Prisma client
3. **Module load time** in `src/middleware/auth.ts` - validates JWT_SECRET for auth

This ensures the app exits immediately if configuration is invalid, preventing silent failures in production.

## Health Check Endpoint

A new `/health` endpoint is available for monitoring:

```bash
curl https://your-backend.up.railway.app/health
```

Response:

```json
{
  "status": "ok",
  "environment": "production",
  "timestamp": "2026-03-23T00:00:00.000Z"
}
```

Use this for:

- Railway health checks (configure in dashboard)
- Uptime monitoring services
- Load balancer checks
